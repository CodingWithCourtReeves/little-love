# v0.2 Worktree Runbook

How to spawn each of the five worktree-paired Claude sessions, in order, with the right hand-off prompts.

## Pre-flight (once)

From the main repo (`~/projects/little-love`):

```bash
git checkout main
git pull --rebase
```

Confirm the spec and all five briefs are present:

```bash
ls docs/superpowers/specs/2026-06-09-littlelove-accounts-and-inbox-design.md
ls docs/superpowers/plans/2026-06-09-v0.2-WT-{A,B,C,D,E}-brief.md
```

If any are missing — stop, re-run the brainstorming/spec session.

## Verification rounds (your driving cadence)

| Round | Parallel WTs | What you verify after both PRs merge |
|---|---|---|
| **R1** | WT-A + WT-C | Signup on Mac → recovery phrase → reopen, Touch ID unlock → signed in. Same phrase signs in on Kaitlyn's Win11 (Windows Hello PIN). Server log shows signed Identify. → Tag `v0.2.0-auth`. |
| **R2** | WT-B + WT-D | Court creates invite → 4-word code in clipboard. Kaitlyn enters it. Both see the conversation. Send "test"; server `psql` returns base64 garbage in `body`. Kaitlyn renders "test". → Tag `v0.2.0-pair`. |
| **R3** | WT-E (solo) | Multiple conversations in sidebar (use a 2nd fixture pairing if needed). Switch is instant. Resize 1400 → 700 → 500; transitions clean at 800 & 600. Tap targets ≥ 44×44. → Tag `v0.2.0-inbox`. |
| **Integration** | one session | Wire WT-D's real `roomProvider` into WT-E's inbox state. End-to-end smoke. → Tag `v0.2.0`. |

If a WT discovers a spec gap mid-implementation, the session **stops**, you amend the spec on `main`, and all open WTs rebase before continuing (spec §12.4).

## Spawning a worktree session

For WT-X (replace `X` with A/B/C/D/E):

```bash
# From the main repo:
git worktree add ../little-love-WT-X -b feat/v0.2-WT-X
cd ../little-love-WT-X

# dev-env.sh will produce per-worktree ports automatically:
source scripts/dev-env.sh
```

Then open a new Claude Code session in `../little-love-WT-X`.

### Opening prompt to paste into that session

```
You are WT-X for LittleLove v0.2 implementation.

Read these files in order (do not skim — these are load-bearing):
1. docs/superpowers/specs/2026-06-09-littlelove-accounts-and-inbox-design.md
   (focus on the sections listed in your brief)
2. docs/superpowers/plans/2026-06-09-v0.2-WT-X-brief.md

Then:
1. Invoke the superpowers:writing-plans skill, scoped strictly to this WT's
   acceptance criteria + files. Save the plan to:
   docs/superpowers/plans/2026-06-09-v0.2-WT-X.md
2. After the plan is written, present it and ask me to confirm before executing.
3. On confirmation, invoke superpowers:subagent-driven-development
   (or superpowers:executing-plans if subagent support is unavailable)
   to execute the plan task-by-task.
4. Commit per the plan; each commit message references its task ID.
5. When all WT acceptance criteria from your brief are met:
   - Run lint + tests locally.
   - Open a PR against main with `gh pr create`.
   - Stop and ping me.

Stop conditions: if you find any spec ambiguity or contract gap, STOP IMMEDIATELY,
do not work around it. Amending the spec on main is the correct response.
```

### Notes per WT

- **WT-A:** Can start as soon as the spec + brief are on `main`. No upstream dependency.
- **WT-B:** Wait for WT-A's PR to be merged. Your tests depend on the new accounts table + auth handshake.
- **WT-C:** Wait for WT-A's PR to be merged. Your client tests will exercise the new server endpoints.
- **WT-D:** Wait for both WT-B and WT-C to be merged. You need WT-B's `server/tests/data/invite_vectors.json` fixture and WT-C's identity layer.
- **WT-E:** Wait for WT-C to be merged. WT-D not blocking (mock the room provider in dev).

### Running R1, R2, R3 in parallel (recommended pace)

You don't have to run all five WTs back-to-back. The healthy rhythm is:

1. Spawn WT-A. When it PRs, spawn WT-C in parallel with WT-B.
2. Once WT-A merges, WT-B and WT-C unblock. They can run concurrently in separate worktrees + sessions.
3. After both R1 verification (you, manually) and merging, spawn WT-D.
4. WT-E can begin in parallel with WT-D using mocked data.
5. R2 verification happens once WT-B + WT-D both merge.
6. R3 verification once WT-E merges.
7. Integration session to glue WT-D's real provider into WT-E's UI; tag v0.2.0.

## Tearing down a worktree

After a WT's PR is merged and verified:

```bash
cd ~/projects/little-love
git worktree remove ../little-love-WT-X
git branch -d feat/v0.2-WT-X
```

(Don't remove the worktree until the PR is fully merged; the branch may still be needed for rebases if the spec is amended.)

---

## Common pitfalls

- **Don't run a WT against `main` directly.** Always create a worktree first. The dev-env scripts compute per-worktree ports from the basename; running on `main` would collide with other sessions' Postgres.
- **Don't skip the "read the spec" step.** The spec is the contract source of truth. Each brief tells you which spec sections matter; ignore the rest at your peril, but those listed ones are non-negotiable.
- **Don't substitute crypto libraries.** Spec §8.5 names exact crates. A workstream that picks a different X25519 lib will produce keys that don't match other workstreams' bytes.
- **Don't widen scope.** If a brief says you don't own a file, you don't touch it. Cross-WT edits cause merge conflicts and integration debt.
- **Don't push through a spec ambiguity.** Stop, ping Court, amend `main` first.
