**Spec:** `docs/superpowers/specs/2026-06-09-littlelove-accounts-and-inbox-design.md`
**Branch:** `feat/v0.2-WT-E`
**Depends on:** WT-C merged (you consume `accountProvider`). WT-D not strictly blocking — you can build against a mocked `roomProvider` during development and integrate at R3.
**Verification round:** R3 — solo. Tag at end: `v0.2.0-inbox`.

## What this WT delivers

The multi-conversation inbox UI: sidebar + detail at desktop widths, NavigationRail at 800px, Drawer at 600px. Per-conversation message state, switching, mobile-first invariants. By the end:
- Signed-in user with rooms sees a sidebar listing them.
- Selecting a room renders that room's conversation in the detail pane.
- Switching conversations is instant; each room has its own message buffer.
- Resizing the window collapses sidebar → rail → drawer at the spec breakpoints with no jank.
- Composer stays anchored, tap targets meet 44×44 minimum.

## Acceptance criteria

You are done when:

1. Spec AC #5 passes:
   - Window at 1400px → sidebar (240px) + detail.
   - Window at 700px → NavigationRail + detail.
   - Window at 500px → Drawer + full-screen detail; hamburger toggles drawer.
   - Resize transitions are smooth (no full re-mount of the conversation page).
2. Selecting a conversation in the sidebar/rail/drawer changes the detail pane's room context within ~50ms.
3. Per-conversation message stores are independent (sending in room A doesn't appear in room B's view).
4. All tap targets (sidebar entry rows, rail icon buttons, drawer entry rows, composer send button) are ≥ 44 × 44 logical px (asserted via `tester.getSize` in widget tests).
5. Composer stays pinned to the bottom of the detail viewport at all widths; scroll the message list above it; auto-scroll behavior from Day-1 is preserved per-room.
6. `dart format --set-exit-if-changed`, `flutter analyze`, `flutter test` all green.

## Stop conditions

- Flutter's `MediaQuery.size` doesn't fire on window resize (would require `WindowResizeNotifier` or similar) — investigate before adding workarounds; the spec assumes standard `MediaQuery` reactivity.
- A `NavigationRail` / `Drawer` material widget behavior contradicts spec §6.4 (mobile-first invariants).
- Spec ambiguity in §6 or §11.
