> **Spec amendment 2026-06-09 (post-brief):** §3.3 step 3, §4.1 step 10, and the new §8.5.1 introduce **domain-separated signing inputs**. Your Ed25519 verify must prepend `b"littlelove.v0.2.challenge" || 0x00` to the nonce before calling `verify_strict`. Use `ed25519-dalek::VerifyingKey::verify_strict` (not `verify`) — it closes weak-key forgery + cofactor malleability. See §8.5.1 for the exact byte layout and a test vector.

## What this WT delivers

Real accounts and authenticated WSS sessions on the server. By the end of this WT:
- `POST /accounts` creates an account from a username + Ed25519 pub + X25519 pub.
- `GET /accounts/by-username/{username}` returns those publics.
- WSS handshake replaces the trusted `x-llove-user` header with Challenge → Identify → Authenticated.
- A client signing with the wrong key gets close code 4001.
- The `Send` frame's `from` field is overridden server-side with the authenticated identity (no spoofing).

## Acceptance criteria

You are done when:

1. Spec AC #1 server-side passes: a client can `POST /accounts`, then complete a `Challenge → Identify → Authenticated` flow over WSS. Wrong signature → close 4001.
2. Spec AC #6 passes: `x-llove-user` header reads are deleted; `Send` frames inbound have `from` rewritten to the authenticated account; tests prove a forged `from` is overridden.
3. `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test -p server` all green.
4. Integration test `server/tests/auth_handshake.rs` covers: success path, bad signature, unknown username, replay attack (server's nonce is single-use per connection).
