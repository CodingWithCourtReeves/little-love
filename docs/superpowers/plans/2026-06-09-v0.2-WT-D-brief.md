## What this WT delivers

The pairing flow on the client + per-conversation ECDH key derivation. By the end:
- Signed-in user without a partner taps "Pair with partner" → app sends `CreateInvite` → 4-word code + QR appear.
- Receiving user pastes the code → app shows `Pair with @<inviter>?` → confirms → app sends `ConsumeInvite` with a signature.
- Both clients receive room metadata, derive the ECDH room key locally, and the conversation appears in the inbox.
- Outbound messages are encrypted with the room key before send; inbound messages are decrypted using the same key.

## Acceptance criteria

You are done when:

1. Spec AC #2 client-side passes: invite create produces a working code; entering a code on the other client successfully consumes it; both clients now have a `Room` in their inbox state.
2. Spec AC #3 client-side passes: messages sent in a room are encrypted with the ECDH-derived key; received messages are decrypted with the same key; failures fall back to "could not decrypt" message text.
3. The BIP39 invite encoding/decoding matches WT-B's fixture file byte-for-byte (test reads `server/tests/data/invite_vectors.json` from the workspace root and asserts identical 4-word codes for the 8 input tokens).
4. ECDH derivation matches a deterministic test vector (use spec §5.1 to derive at least 2 vectors with known X25519 keypair pairs and assert the resulting room keys match).
5. `dart format --set-exit-if-changed`, `flutter analyze`, `flutter test` all green.
