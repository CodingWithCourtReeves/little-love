# LittleLove v0.3 — Bot Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring the `littlelove-bot` binary forward to v0.3 wire compatibility (Send fan-out, multi-room subscribe, RoomCreated/RoomRenamed/MemberLeft frames) and add a new owner-derived registration path (`littlelove-bot spawn --label <label>`) that uses the v0.3 `POST /accounts/bot` endpoint with a domain-separated owner signature. PR #8 bots (with their own master secret in `identity.json`) keep working unchanged via the existing `pair` subcommand — see spec §10.3.

**Architecture:** A new pure-crypto helper `derive_bot_identity(owner_master, label)` lives in `littlelove-crypto` so both the bot binary and any future server-side validator can re-derive the bot's Ed25519/X25519 pubkeys from the owner's master seed plus a free-form label. The bot's `spawn` subcommand reads the owner's master secret from `--owner-identity-file <path>` (the same `identity.json` format the v0.2 client writes), derives the bot identity, calls `POST /accounts/bot` with a signature from the owner's Ed25519 key, and persists a v0.3-flavored `identity.json` that stores the **bot_label** rather than a fresh master secret. The bot's run-loop tracks **N rooms** (a HashMap of room_id → derived room keys per peer pubkey) and on `Send` builds a `bodies` map.

**Tech Stack:** Rust 1.88, the existing bot crate's deps (`tokio-tungstenite`, `clap`, `reqwest`, `serde_json`, `ed25519-dalek`, `hkdf`, `sha2`, `chacha20poly1305`, `x25519-dalek`), and the shared `littlelove-crypto` workspace member.

**Out of scope:** Migrating v0.2 PR #8 bots to owner-derived identity (`reseed --derive-from-owner` is a future revision; spec §10.3 keeps existing bots as-is). Session locking for two-process bots (spec §4.3). Reaching beyond the local network for the LLM endpoint — `addr_guard` still enforces the private-IP allow-list. UI for owner-facing bot management (covered by the client plan's Create Chat picker + `DELETE /accounts/bot/{label}`).

**Spec sections implemented:** §3.3 (familiar definition + derived identity), §4.1 (registration), §4.2 (operation), §4.3 (multi-device documented limitation), §6.x (fan-out send + per-recipient receive), §7.4 (handle MemberLeft), §10.3 (v0.2 bots untouched).

## Conventions & Test Infrastructure

- **`bot/tests/spawn_register.rs`** uses a mocked HTTP server (`wiremock` is already in workspace) to assert the POST body shape without booting a real littlelove-api process. We do NOT add wiremock as a new dep — if it's not already in `bot/Cargo.toml` dev-deps, the test uses a small `axum` mini-server stood up inside the test (existing pattern from the v0.2 bot tests).
- **`bot/tests/fanout_reply.rs`** uses a fake WSS Server identical to the v0.2 bot test harness (`common::run_fake_ws`) — extend it to accept v0.3 Send.bodies and v0.3 Subscribe.
- **`derive_bot_identity` is pure-crypto** — its test lives in `crypto/src/identity.rs` and runs as part of `cargo test -p littlelove-crypto`.
- **PR #8 bots' `identity.json`** uses version 1. The `spawn` subcommand writes version 2 with `bot_label` and `owner_username` set. The `run` subcommand reads either version and derives the keypair correctly per the file's version field.
- **No backwards-compat shim on the wire.** A v0.2 bot binary won't be able to send to a v0.3 server (Send.body is gone). v0.2 bots must be rebuilt at the new revision and rerun against their existing identity.json. Court's runbook covers this.
- **Commit cadence:** one commit per task. Required checks: `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`.

const SALT_BOT: &[u8] = b"littlelove.v0.3.bot";

/// Spec §3.3: derive a familiar's keypair deterministically from its owner's
/// master seed + a free-form bot label. Same `(owner_master, label)` always
/// reproduces the same Ed25519 + X25519 keypair.
pub fn derive_bot_identity(owner_master: &[u8; 32], label: &str) -> Result<Identity, IdentityError> {
    let info = {
        let mut v = Vec::with_capacity(4 + label.len());
        v.extend_from_slice(b"bot:");
        v.extend_from_slice(label.as_bytes());
        v
    };
    let bot_seed = hkdf_extract_expand(SALT_BOT, owner_master, &info)?;
    derive_identity(&bot_seed)
}
```

#[derive(Debug, Serialize)]
pub struct RegisterBotRequest {
    pub owner_username:   String,
    pub bot_label:        String,
    pub bot_username:     String,
    pub bot_ed25519_pub:  String,    // base64
    pub bot_x25519_pub:   String,    // base64
    pub owner_signature:  String,    // base64
}

#[derive(Debug, Deserialize)]
pub struct RegisterBotResponse {
    pub account_id:   i64,
    pub bot_username: String,
    pub is_bot:       bool,
}

pub async fn register_bot(base_url: &str, req: &RegisterBotRequest) -> Result<RegisterBotResponse> {
    let url = format!("{}/accounts/bot", base_url.trim_end_matches('/'));
    let resp = reqwest::Client::new()
        .post(&url)
        .json(req)
        .send()
        .await
        .with_context(|| format!("POST {url}"))?;
    let status = resp.status();
    if !status.is_success() {
        let body = resp.text().await.unwrap_or_default();
        return Err(anyhow!("register_bot {status}: {body}"));
    }
    let parsed: serde_json::Value = resp.json().await?;
    // Server returns "account_id"/"bot_username"/"is_bot".
    Ok(RegisterBotResponse {
        account_id: parsed["account_id"].as_i64().unwrap_or(0),
        bot_username: parsed["bot_username"].as_str().unwrap_or("").to_string(),
        is_bot: parsed["is_bot"].as_bool().unwrap_or(true),
    })
}
```

`bot/src/spawn.rs`:

```rust
//! spawn subcommand: derive a bot identity from the owner's master seed,
//! register it via POST /accounts/bot, persist a v0.3 IdentityFile.

use anyhow::{anyhow, Context, Result};
use base64::{engine::general_purpose::STANDARD as B64, Engine};
use chrono::Utc;
use ed25519_dalek::Signer;

use crate::cli::SpawnArgs;
use crate::identity_store::{default_identity_path, load_identity, save_identity, IdentityFile};
use crate::rest::{register_bot, RegisterBotRequest};
use littlelove_crypto::identity::{derive_bot_identity, derive_identity};
use littlelove_crypto::sig::bot_register_signing_input;

fn label_ok(s: &str) -> bool {
    (1..=32).contains(&s.len())
        && s.bytes().all(|b| b.is_ascii_lowercase() || b.is_ascii_digit() || b == b'-')
}

pub async fn run(args: SpawnArgs) -> Result<()> {
    if !label_ok(&args.label) { anyhow::bail!("--label must match [a-z0-9-]{{1,32}}"); }

    let bot_path = std::env::var("LITTLELOVE_BOT_IDENTITY_PATH")
        .map(std::path::PathBuf::from)
        .unwrap_or_else(|_| default_identity_path());
    if bot_path.exists() && !args.force {
        anyhow::bail!("identity file already exists at {bot_path:?} — pass --force to overwrite");
    }

    let owner = load_identity(&args.owner_identity_file)
        .with_context(|| format!("read owner identity {:?}", args.owner_identity_file))?;
    let owner_master_bytes = B64.decode(owner.master_secret_b64.as_bytes())
        .context("decode owner master_secret_b64")?;
    let owner_master: [u8; 32] = owner_master_bytes.as_slice().try_into()
        .map_err(|_| anyhow!("owner master_secret_b64 not 32 bytes"))?;

    // Owner's signing key, re-derived from master.
    let owner_id = derive_identity(&owner_master)?;

    // Bot identity (deterministic from owner_master + label).
    let bot_id = derive_bot_identity(&owner_master, &args.label)?;
    let bot_ed = bot_id.ed25519_pub();
    let bot_x  = bot_id.x25519_pub();

    let bot_username = args.bot_username.unwrap_or_else(|| format!("{}-{}", owner.username, args.label));

    let sig = owner_id.ed25519_signing.sign(&bot_register_signing_input(&bot_ed)).to_bytes();
    let req = RegisterBotRequest {
        owner_username: owner.username.clone(),
        bot_label:      args.label.clone(),
        bot_username:   bot_username.clone(),
        bot_ed25519_pub: B64.encode(bot_ed),
        bot_x25519_pub:  B64.encode(bot_x),
        owner_signature: B64.encode(sig),
    };
    let rest_base = ws_to_rest_base(&args.server)?;
    let resp = register_bot(&rest_base, &req).await.context("register_bot")?;

    // Persist v2 identity. master_secret_b64 is left empty for v2 bots —
    // the bot rederives on every start from owner master + label, NOT from
    // its own master. (`run` reads bot_label + owner_username + owner master
    // from disk and re-derives at boot.)
    let file = IdentityFile {
        version: 2,
        username: resp.bot_username,
        ed25519_pub_b64: B64.encode(bot_ed),
        x25519_pub_b64:  B64.encode(bot_x),
        master_secret_b64: String::new(),
        bot_label: Some(args.label),
        owner_username: Some(owner.username),
        created_at: Utc::now(),
    };
    save_identity(&bot_path, &file, args.force).context("save bot identity")?;

    println!("Familiar @{} registered. Identity at {}.",
        file.username, bot_path.display());
    Ok(())
}

fn ws_to_rest_base(server: &str) -> Result<String> {
    let s = server.trim_end_matches('/');
    if let Some(r) = s.strip_prefix("wss://") { return Ok(format!("https://{r}")); }
    if let Some(r) = s.strip_prefix("ws://")  { return Ok(format!("http://{r}"));  }
    Ok(s.to_string())
}
```

- [ ] **Step 3: Important — the owner master secret is on disk.**

A v0.3-style bot rederives its keypair from the owner's master seed on every start. That requires the owner's `identity.json` to be reachable to the bot process. **This is a deployment posture decision Court should confirm:**

- Option A (what this plan ships): bot reads `owner_identity_file` path from its own identity.json (`owner_identity_file` stored in IdentityFile v2 as a new field). Pro: zero per-start interaction. Con: owner master secret sits unencrypted at the configured path; if the bot host is compromised, the owner identity is leaked.
- Option B (more secure): bot persists its OWN bot-keypair bytes in the bot's identity.json (the same way PR #8 v0.2 bots do), keyed by `bot_label`. The `spawn` flow registers with POST /accounts/bot but then writes a derived-and-frozen keypair to disk. Owner master leaves the spawn process and is never read again. **This is the recommended approach** — spawn is a one-shot.

Let's pick **Option B** for v0.3.0. Edit Step 2's implementation: after deriving `bot_id`, encode `bot_id.master` (the bot's own intermediate seed produced by derive_bot_identity) into the IdentityFile `master_secret_b64`, marking version 2 to distinguish from v0.2 bots. The `bot_label` and `owner_username` stay alongside for diagnostic/UX purposes.
