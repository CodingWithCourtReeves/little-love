## What this WT delivers

Invites, rooms, room-membership routing on the server. By the end:
- Authenticated client can `CreateInvite` and get back a 4-word code + QR.
- Unauthenticated REST `POST /invites/{code}/preview` returns inviter publics.
- Authenticated client can `ConsumeInvite` with a signature; server creates a room with both members.
- WSS routes `Send` frames by room membership; replays history per-room on `Subscribe`.
- Monogamy is enforced: an account can be in at most one room.
- Multi-session fan-out: a user with two open WSS sessions sees every message at both.

## Acceptance criteria

You are done when:

1. Spec AC #2 server-side passes: invite create → preview → consume produces a room with both members; `RoomCreated` is pushed to the inviter's connected sessions.
2. Spec AC #3 server-side passes: messages are stored with their wire body bytes unchanged; `psql` `SELECT body FROM messages` returns base64 garbage (you don't decrypt anything server-side — you just enforce ciphertext flows opaquely).
3. Spec AC #4 passes: two WSS connections per username receive every message routed to that room.
4. Monogamy enforcement: a second `ConsumeInvite` from an already-paired account returns `Error { code: "AlreadyPaired" }`.
5. BIP39 encoding fixture file `server/tests/data/invite_vectors.json` is committed and read by `bip39_invite_codes.rs`. The fixture lists 8 input tokens (32-byte hex) and their canonical 4-word codes per spec §8.6.
6. `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test -p server` all green.

## Crypto / encoding primitives

- `ulid = "1"` for `room_id` / `message_id`.
- `sha2 = "0.10"` for `token_hash`.
- `qrcode = "0.14"` + `image = "0.25"` for QR PNG generation (output base64-encoded inline).
- BIP39 wordlist: vendor the English 2048-word list in `server/src/wordlist_bip39_en.rs` as a `&[&str; 2048]` constant. Source: BIP39 official spec.
