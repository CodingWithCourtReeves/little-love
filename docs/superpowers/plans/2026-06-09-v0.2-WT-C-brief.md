> **Spec amendment 2026-06-09 (post-brief):** §3.3 step 3 and the new §8.5.1 introduce **domain-separated signing inputs**. When responding to a `Challenge` frame, your client must sign `b"littlelove.v0.2.challenge" || 0x00 || nonce` (58 bytes total), NOT the bare nonce. The base64-encoded signature still goes into the `Identify.signature` field unchanged. See §8.5.1 for the exact byte layout and a test vector. If you've already drafted the signer, this is a one-line prepend before calling Ed25519 sign.

## What this WT delivers

Real signup, recovery-phrase show + confirm, sign-in, and authenticated WSS handshake on the client. By the end:
- Fresh app launch with no `account.json` → signup screen.
- User enters username, generates a 12-word recovery phrase, confirms by re-typing words 3/7/11, account is created server-side and keypair is sealed in OS keystore.
- App restart → OS keystore unlock (Touch ID on Mac, Windows Hello PIN/biometric on Win11) → app signed in.
- Manual "sign in with recovery phrase" screen restores identity on a fresh device.
- WSS connection authenticates via Challenge → Identify → Authenticated.

## Acceptance criteria

You are done when:

1. Spec AC #1 client-side passes:
   - Fresh launch with no `account.json` → signup screen renders.
   - Entering a valid username completes signup: phrase shown once, confirmation step works, account is POSTed and stored locally.
   - App restart triggers OS keystore unlock and lands in signed-in state.
   - Manual "sign in with phrase" screen restores identity correctly.
2. WSS connection authenticates via Challenge → Identify → Authenticated. A wrong signature produces a recognizable error in the UI ("could not authenticate, recovery phrase may be wrong").
3. `dart format --set-exit-if-changed`, `flutter analyze`, `flutter test` all green.
4. Widget tests assert: signup widget shows exactly 12 words; confirmation requires words 3/7/11 in order; mismatch → reject.
