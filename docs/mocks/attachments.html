How the preview thumbnail works (and why it's instant): when you send a photo/video, the client makes a tiny downscaled JPEG (≤180px, a video poster frame for clips), encrypts it, and tucks it inside the message body itself — the same per-recipient encrypted envelope that already carries text. So the recipient renders the preview the moment the message arrives, with no extra download. The full-resolution file is encrypted once with its own key, uploaded to R2, and only fetched + decrypted when you tap to open (screen 6). The server brokers short-lived upload/download URLs over the authenticated WebSocket but never sees the file key — R2 holds only ciphertext. Status marks match the existing message-status spec: a faint clock while uploading/sending, a heart once the server acks. Thumbnails here are CSS gradients standing in for real photos.
